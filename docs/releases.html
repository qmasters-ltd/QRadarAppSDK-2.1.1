QRadar App SDK - release notes

2.1.0

  • App registration with new actions preregister, register, and deregister.
    See QRadar App SDK - app registration.
  • The run action has a new -q option for associating a running container with a QRadar server.
  • For any server-related action, the -q and -u options are no longer mandatory.
    The SDK stores the most recent server and user values you supplied, and subsequently uses them as default values when you omit the -q or -u options.
  • The new server action shows the -qand -u default values for server-related actions, and checks the certificate bundle status for the server.
  • The new qapp -d option displays SDK documentation.
  • When an app is run locally, a container memory limit is now set. The memory limit used is the resources.memory value in the app workspace’s manifest.json file, and defaults to 100MB if not supplied.
  • The uuid field in the app workspace’s manifest.json file is now validated to ensure that it conforms to the UUID format.
  • The example app’s container/README.txt file has updated information about dependency handling.
  • App base image updated to v2.1.6: security updates, qpylib v2.0.6.
  • qjslib updated to v1.2.2: security updates, no functional changes.
  • SDK documentation restructure.

2.0.2

  • Resolve SDK install script failure caused by Python package dependency cffi.
    The issue occurs only when your Python environment is at version 3.9.1 or later, and is resolved by updating the cffi version from 1.14.2 to 1.14.4.

2.0.1

  • Resolve Connection broken: IncompleteRead(0 bytes read) issue with qapp run and qapp clean.
    The issue can occur on MacOS or Windows, and is specific to Docker Desktop v2.5+.

2.0.0

  • SDK reworked to support QRadar App Framework v2 and Python 3.
  • To execute the SDK v2 install script, root privileges are no longer required.
  • Python 3 is required to install and run the SDK.
  • Apps that include Python code in their implementation must use Python 3.
  • SDK v2 actions are invoked using the command qapp rather than the v1 entry point qradar_app_creator.
  • Using Docker, SDK v2 supports building an app image and running an app container locally. This functionality replaces the SDK v1 mechanism, which executed app code locally within a Python virtual environment.
  • App images are built using a base image derived from RHEL8 Universal Base Image (UBI).
  • Inside the container, all app-related files and directories are now located under /opt/app-root rather than the root directory /.
  • Processes within the app container now run as a non-privileged user, not as root. The same non-privileged user owns all app-related files and directories inside the container.
  • The src_deps directory for defining app dependencies and scripts has been replaced by a directory named container. For more details see the create and build actions.
  • To deploy a UBI-based app to a QRadar server, the manifest.json file must contain an image entry.
  • The SDK’s template app has been restructured to better reflect Python/Flask recommended practices such as using an application factory and defining routes using blueprints.
  • The source code for qpylib is no longer bundled with the other app source files. Instead, qpylib is pre-installed as a Python package in the app base image.
  • The first invocation of qapp each day performs a check to determine if the installed SDK version is the latest.
  • SDK v2 has a new mechanism for server certificate retrieval and SOCKS proxy configuration.
